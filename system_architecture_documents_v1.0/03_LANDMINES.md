# 03_LANDMINES.md

**Top 12 Interview Landmines – Evidence‑Based**

| # | Title | Severity | What Exists | What Is Missing | How Interviewer May Detect | Honest Explanation | Professional Framing | Production Fix |
|---|-------|----------|-------------|----------------|---------------------------|--------------------|----------------------|----------------|
| 1 | Runtime vs README Mismatch | High | Code implements `/chat`, `/health`, `/metrics`. | README claims Kubernetes auto‑scaling and CI/CD. | Interviewer asks about K8s deployment; code shows no orchestration. | The repo is a demo; deployment scripts are placeholders. | Acknowledge the gap and outline planned Helm charts and GitHub Actions. | Add CI pipeline, Helm chart, and configure readiness probes. |
| 2 | Dead Module – `app.tools.order_status` Mock | Medium | Mock tool present and wired in workflow. | Real order‑status integration. | Interviewer probes tool’s external calls. | Mock used for interview demo to avoid external dependencies. | Explain it demonstrates intent handling; real integration would be added later. | Implement real API client with retries and circuit‑breaker. |
| 3 | Missing Loki Integration | Medium | Loki config file exists. | No exporter wired in observability module. | Interviewer asks about log aggregation. | Loki was scoped for future observability expansion. | State it’s planned; current logs go to stdout for simplicity. | Wire `opentelemetry‑exporter‑loki` and send logs. |
| 4 | In‑Memory Session State | High | `app.agent.memory` stores sessions in a dict. | Persistence across restarts. | Interviewer inquires about crash recovery. | Chosen for demo simplicity; not production‑ready. | Highlight need for external store. | Move session store to Redis with TTL. |
| 5 | SQLite Persistence Limitation | Medium | SQLite DB used via SQLAlchemy. | Scalability & concurrent writes. | Interviewer asks about high traffic handling. | SQLite suffices for low‑volume demo. | Discuss migration path. | Switch to PostgreSQL with Alembic migrations. |
| 6 | Unwired Deployment Artifacts | High | `Dockerfile` and `docker‑compose.yml` present. | No code triggers container orchestration; no CI/CD. | Interviewer asks about container rollout. | Artifacts are scaffolding for future production. | Emphasize they enable quick containerisation when needed. | Add entrypoint scripts, multi‑stage builds, and CI pipeline. |
| 7 | No Authentication / Authorization | High | All endpoints are open. | Auth layer (OAuth2/JWT). | Interviewer queries security model. | Demo focuses on core functionality; auth omitted. | Acknowledge security is a next step. | Integrate FastAPI security utilities with JWT. |
| 8 | Limited Observability Metrics | Medium | Basic request latency via `Timer`. | Detailed histograms, error counters, tracing. | Interviewer requests metric granularity. | Initial metrics cover health; deeper insights planned. | Outline roadmap for OpenTelemetry tracing. | Add Prometheus histograms, error counters, and trace exporters. |
| 9 | Vector Store Durability | Medium | Chroma DB in‑memory/SQLite directory. | Replication, backup, scaling. | Interviewer asks about vector store resilience. | Chosen for simplicity; not production‑grade. | Mention evaluation of managed vector DBs. | Deploy Milvus or Pinecone with HA. |
|10| No Rate Limiting / Throttling | Medium | No middleware limiting requests. | Rate‑limit implementation. | Interviewer probes abuse protection. | Demo environment assumes trusted users. | Plan to add `slowapi` or similar. | Implement per‑IP rate limiting and burst control. |
|11| Absence of CI/CD Pipelines | High | No `.github/workflows/`. | Automated testing, linting, deployment. | Interviewer expects CI badges. | CI was out of scope for prototype. | Commit to adding GitHub Actions. | Add CI workflow with pytest, flake8, Docker build. |
|12| Scalability Bottleneck – Single‑process Memory | High | All components run in single FastAPI process. | Horizontal scaling, load balancer. | Interviewer asks about scaling to many users. | Prototype runs single worker for simplicity. | Discuss using Uvicorn workers and container orchestration. | Deploy with Gunicorn workers, Kubernetes HPA. |

*All items are directly traceable to the audited evidence files; no speculative features are presented.*
